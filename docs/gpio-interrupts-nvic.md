# GPIO Interrupts & NVIC

Many embedded systems spend most of their time waiting for external events.

Instead of constantly polling GPIO pins in a software loop, STM32
microcontrollers can generate hardware interrupts whenever a configured signal
changes.

Several projects in this repository rely on this mechanism:

- Stopwatch push buttons
- Fan tachometer measurement
- Joystick input

Using interrupts allows the CPU to remain idle until something actually happens.

---

# Polling vs Interrupts

There are two common ways to detect an external event.

## Polling

```text
while (1)
{
    read GPIO

    event?

    read GPIO

    event?

    read GPIO
}
```

The CPU continuously checks the input, even when nothing changes.

Advantages

- very simple
- easy to debug

Disadvantages

- wastes CPU time
- slower reaction for long polling intervals

---

## Interrupts

```text
External signal

        │

        ▼

GPIO detects edge

        │

        ▼

EXTI generates interrupt

        │

        ▼

NVIC

        │

        ▼

Interrupt Service Routine (ISR)
```

The CPU continues executing other code—or sleeps entirely—until the hardware
detects the configured event.

Interrupts therefore provide

- faster response
- lower CPU usage
- deterministic event handling

---

# EXTI (External Interrupt Controller)

GPIO interrupts are handled by the STM32 EXTI peripheral.

Each EXTI line can be connected to one GPIO pin.

```text
GPIO Pin

      │

      ▼

EXTI Line

      │

      ▼

NVIC

      │

      ▼

ISR
```

Whenever the selected edge occurs, EXTI requests an interrupt from the NVIC.

---

# Trigger Modes

STM32 supports three trigger modes.

| Mode | Trigger |
|------|----------|
| `GPIO_MODE_IT_RISING` | Low -> High |
| `GPIO_MODE_IT_FALLING` | High -> Low |
| `GPIO_MODE_IT_RISING_FALLING` | Both edges |

Example:

```c
GPIO_InitTypeDef gpio = {
    .Pin  = GPIO_PIN_7,
    .Mode = GPIO_MODE_IT_RISING,
    .Pull = GPIO_NOPULL
};

HAL_GPIO_Init(GPIOB, &gpio);
```

The Fan Controller uses a **rising-edge interrupt** to detect each tachometer
pulse generated by the fan.

---

# Interrupt Processing

Once an interrupt occurs, STM32 automatically performs several hardware steps.

```text
GPIO Edge

      │

      ▼

Current instruction finishes

      │

      ▼

CPU saves context

      │

      ▼

Jump to ISR

      │

      ▼

Execute callback

      │

      ▼

Restore context

      │

      ▼

Continue previous code
```

Because the CPU state is restored automatically, the interrupted program
continues exactly where it stopped.

---

# HAL Callback

Most GPIO interrupts ultimately arrive in

```c
HAL_GPIO_EXTI_Callback()
```

Example:

```c
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin)
{
    if (GPIO_Pin == TACHO_PIN)
    {
        /* Measure RPM */
    }
}
```

Only minimal processing should be performed inside an interrupt.

Long calculations should instead be deferred to the main application.

---

# NVIC (Nested Vectored Interrupt Controller)

The NVIC is responsible for scheduling interrupts.

Its tasks include

- enabling or disabling interrupt sources
- assigning priorities
- deciding which interrupt executes first
- allowing interrupt nesting

Typical configuration:

```c
HAL_NVIC_SetPriority(
    EXTI9_5_IRQn,
    0,
    0);

HAL_NVIC_EnableIRQ(
    EXTI9_5_IRQn);
```

---

# Interrupt Priorities

When several interrupts occur simultaneously, the NVIC selects the one with the
highest priority.

```text
Priority 0

Highest

↓

Priority 1

↓

Priority 2

↓

Priority 15

Lowest
```

Choosing sensible priorities is important in real-time systems.

For example, a motor-control interrupt often deserves a higher priority than a
user-interface button.

---

# Interrupt vs Event

EXTI supports both **Interrupt** and **Event** modes.

| Interrupt | Event |
|-----------|-------|
| Executes software (ISR) | No ISR |
| CPU reacts | Hardware reacts |
| Changes program flow | Does not interrupt software |

Events are mainly intended for peripheral-to-peripheral synchronization without
CPU involvement.

The projects in this repository exclusively use **Interrupt Mode**.

---

# Why EXTI Has 23 Lines

A GPIO port only contains sixteen pins, yet STM32 exposes twenty-three EXTI
lines.

```
EXTI 0–15

↓

GPIO pins

EXTI 16–22

↓

Internal peripherals
```

The additional lines are connected to internal events such as

- RTC alarms
- Power Voltage Detector (PVD)
- USB wake-up
- Ethernet wake-up

rather than external GPIO pins.

---

# USE_HAL_TIM_REGISTER_CALLBACKS

Timer interrupts normally share a single callback:

```c
HAL_TIM_PeriodElapsedCallback()
```

When several timers are active, the application usually checks

```c
htim->Instance
```

to determine which timer generated the interrupt.

Enabling

```c
USE_HAL_TIM_REGISTER_CALLBACKS
```

allows each timer to register its own callback function instead.

This can simplify larger applications by avoiding a central dispatch function.

---

# Design Takeaways

The projects in this repository illustrate several important real-time design
principles.

- Prefer interrupts over polling for asynchronous events.
- Keep interrupt service routines as short as possible.
- Perform time-consuming work outside the ISR.
- Assign interrupt priorities according to system requirements.
- Let dedicated hardware detect events whenever possible.

Using interrupts effectively is one of the key techniques for building
responsive and efficient embedded systems.
